# PingPulse Rust Client — Design Spec

**Date:** 2026-03-17
**Status:** Approved
**Parent spec:** `docs/superpowers/specs/2026-03-17-pingpulse-design.md`

## Overview

A Rust daemon (`pingpulse`) that maintains a persistent WebSocket connection to the PingPulse Cloudflare Worker backend. It responds to CF-initiated pings, sends its own client-to-CF pings, executes speed tests on demand, and logs all events locally as structured JSON.

## CLI Interface

Binary name: `pingpulse`

```
pingpulse register --token <TOKEN> --name <NAME> --location <LOCATION> --server <URL>
pingpulse start [--foreground]
pingpulse stop
pingpulse status
```

### Commands

**register** — Exchange a registration token for client credentials.
- `--token` (required): Single-use token generated by admin in dashboard
- `--name` (required): Human-readable client name (e.g., "Home Office")
- `--location` (required): Location label (e.g., "Toronto, CA")
- `--server` (required): Base URL of the PingPulse backend (e.g., `https://ping.beric.ca`)
- Sends POST to `{server}/api/auth/register` with `{ token, name, location }`
- Receives `{ client_id, client_secret, ws_url }`
- Writes credentials + config to `~/.pingpulse/config.toml`
- Prints success message with client ID

**start** — Start the daemon.
- Default: installs as a system service (launchd/systemd/sc.exe) and starts it
- `--foreground`: run in the current terminal (for debugging)
- Reads config from `~/.pingpulse/config.toml`
- Connects WebSocket, enters main event loop

**stop** — Stop the daemon.
- macOS: `launchctl unload ~/Library/LaunchAgents/ca.beric.pingpulse.plist`
- Linux: `systemctl --user stop pingpulse`
- Windows: `sc.exe stop pingpulse`

**status** — Print daemon status.
- Checks if the service is running
- Prints: running/stopped, last seen time from log, uptime

## Configuration

**Location:** `~/.pingpulse/config.toml`

```toml
[server]
base_url = "https://ping.beric.ca"
ws_url = "/ws/abc123"
client_id = "abc123"
client_secret = "secret-value"

[ping]
interval_s = 30
grace_period_s = 60

[speed_test]
probe_size_bytes = 262144
full_test_payload_bytes = 10485760
full_test_schedule = "0 */6 * * *"

[alerts]
latency_threshold_ms = 100
loss_threshold_pct = 5.0

[logging]
level = "info"              # debug, info, warn, error
retention_days = 30
```

Config is initialized at registration with defaults. The `[ping]`, `[speed_test]`, and `[alerts]` sections are updated when the server pushes a `config_update` message over WebSocket.

### RemoteConfig Fields (from backend `ClientConfig`)

The `config_update` message carries all 7 fields. The client must deserialize and persist all of them:

| Field | Config Section | Description |
|-------|---------------|-------------|
| `ping_interval_s` | `[ping]` | Seconds between client-to-CF pings |
| `probe_size_bytes` | `[speed_test]` | Payload size for probe tests |
| `full_test_payload_bytes` | `[speed_test]` | Payload size for full tests |
| `full_test_schedule` | `[speed_test]` | Cron expression for autonomous full tests (e.g., `"0 */6 * * *"`) |
| `alert_latency_threshold_ms` | `[alerts]` | Latency threshold for warnings |
| `alert_loss_threshold_pct` | `[alerts]` | Packet loss threshold for warnings |
| `grace_period_s` | `[ping]` | Seconds before marking client as "down" after disconnect |

### Autonomous Full Test Scheduling

The `full_test_schedule` field is a cron expression. The client does NOT need to parse or schedule cron-based tests — full tests are triggered by the server sending `start_speed_test` over WebSocket (the cron worker on CF handles scheduling). The client stores this field for display/logging purposes only.

## WebSocket Protocol

The client communicates with its assigned Durable Object via WebSocket. Messages are JSON-encoded.

### Message Types (matching `WSMessage` in worker/src/types.ts)

```rust
enum IncomingMessage {
    Ping { id: String, ts: u64, payload: Option<Vec<u8>> },
    /// All 7 fields from backend ClientConfig:
    /// ping_interval_s, probe_size_bytes, full_test_schedule,
    /// full_test_payload_bytes, alert_latency_threshold_ms,
    /// alert_loss_threshold_pct, grace_period_s
    ConfigUpdate { config: RemoteConfig },
    StartSpeedTest { test_type: SpeedTestType },  // "probe" | "full"
}

enum OutgoingMessage {
    /// Reply to a CF ping. `ts` echoes the original ping's `ts` (CF's wall-clock).
    /// `client_ts` is the client's wall-clock at time of reply (ms since epoch).
    Pong { id: String, ts: u64, client_ts: u64 },
    /// Client-initiated ping to CF. `ts` is the client's wall-clock (ms since epoch).
    /// No `payload` field needed — CF echoes back as pong with `client_ts`.
    Ping { id: String, ts: u64 },
    SpeedTestResult { result: SpeedTestResult },
    Error { message: String },
}
```

### Field Semantics

- **Pong.ts**: Echo of the incoming ping's `ts` field (CF's timestamp). Do NOT replace with client's time.
- **Pong.client_ts**: Client's current wall-clock time in ms since epoch.
- **Ping.ts**: Client's current wall-clock time in ms since epoch. CF echoes this back in its pong.
- The `direction` field (`cf_to_client` / `client_to_cf`) is set by the DO when recording results — the client does not send it.

### Connection Flow

1. Build WebSocket URL: `wss://{base_url_host}{ws_url}` (e.g., `wss://ping.beric.ca/ws/abc123`)
2. Set `Authorization: Bearer {client_secret}` header on upgrade request
3. On connect: log event, reset backoff
4. On message: dispatch to handler based on `type` field
5. On disconnect: log event, start reconnect with exponential backoff

### Reconnect Strategy

- Initial delay: 1 second
- Multiplier: 2x
- Max delay: 60 seconds
- Jitter: ±25% of current delay
- Reset to 1s on successful connection

## Main Event Loop

A single `tokio::select!` loop handles all concurrent concerns:

```rust
loop {
    tokio::select! {
        // Incoming WebSocket message
        msg = ws_stream.next() => { ... }

        // Client-to-CF ping tick
        _ = ping_interval.tick() => {
            send_ping(&mut ws_sink).await;
        }

        // Graceful shutdown signal (SIGTERM/SIGINT)
        _ = shutdown_signal() => {
            break;
        }
    }
}
```

### Message Handlers

**Ping from CF** → Immediately reply with pong (include `client_ts: now`). Log the ping.

**Config update** → Write new values to `config.toml`. Update the ping interval timer if it changed. Log the update.

**Start speed test (probe)** → Spawn a task that:
1. Downloads probe payload via HTTP GET `/api/speedtest/download?size={probe_size_bytes}`
2. Uploads same payload via HTTP POST `/api/speedtest/upload`
3. Measures download/upload Mbps and duration
4. Sends `speed_test_result` over WebSocket

**Start speed test (full)** → Spawn a task that:
1. Opens 4 parallel HTTP connections for download (`/api/speedtest/download?size={full_test_payload_bytes / 4}`)
2. Measures aggregate download Mbps
3. Opens 4 parallel HTTP connections for upload
4. Measures aggregate upload Mbps
5. Sends `speed_test_result` over WebSocket

## Speed Test Implementation

### Probe Test
- Single HTTP GET + POST, sequential
- Small payload (default 256KB)
- Measures single-connection throughput

### Full Test
- 4 parallel HTTP streams for download, then 4 for upload
- Large payload (default 10MB, split across connections)
- Measures aggregate throughput
- Rate limited server-side (1 per client per 5 min)

### Result Message
```rust
struct SpeedTestResult {
    client_id: String,
    timestamp: String,        // ISO 8601
    r#type: SpeedTestType,    // "probe" | "full"
    download_mbps: f64,
    upload_mbps: f64,
    payload_bytes: u64,
    duration_ms: u64,
}
```

## Logging

**Location:** `~/.pingpulse/logs/`

**Format:** JSON lines, one per event
```json
{"ts":"2026-03-17T12:00:00-04:00","level":"info","event":"ping_reply","rtt_ms":23.4,"jitter_ms":1.2}
{"ts":"2026-03-17T12:00:30-04:00","level":"info","event":"ws_connected"}
{"ts":"2026-03-17T12:01:00-04:00","level":"info","event":"speed_test_complete","type":"probe","download_mbps":95.2,"upload_mbps":42.1}
```

**File naming:** `pingpulse-YYYY-MM-DD.log`

**Events:**
- `ws_connected`, `ws_disconnected`, `ws_reconnecting`
- `ping_sent`, `ping_reply` (with rtt_ms, jitter_ms)
- `config_updated` (with changed fields)
- `speed_test_start`, `speed_test_complete` (with results)
- `error` (with message)

**Rotation:** New file each day (based on local time). On startup, delete log files older than `retention_days`.

**Implementation:** Custom `tracing` layer that writes to the daily log file. The layer handles file rotation by checking the date on each write and opening a new file when the date changes.

## Service Installation

### macOS (launchd)
Generate `~/Library/LaunchAgents/ca.beric.pingpulse.plist`:
```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "...">
<plist version="1.0">
<dict>
    <key>Label</key><string>ca.beric.pingpulse</string>
    <key>ProgramArguments</key>
    <array>
        <string>/path/to/pingpulse</string>
        <string>start</string>
        <string>--foreground</string>
    </array>
    <key>RunAtLoad</key><true/>
    <key>KeepAlive</key><true/>
    <key>StandardOutPath</key><string>~/.pingpulse/logs/stdout.log</string>
    <key>StandardErrorPath</key><string>~/.pingpulse/logs/stderr.log</string>
</dict>
</plist>
```
Then: `launchctl load ~/Library/LaunchAgents/ca.beric.pingpulse.plist`

### Linux (systemd)
Generate `~/.config/systemd/user/pingpulse.service`:
```ini
[Unit]
Description=PingPulse Network Monitor
After=network-online.target

[Service]
ExecStart=/path/to/pingpulse start --foreground
Restart=always
RestartSec=5

[Install]
WantedBy=default.target
```
Then: `systemctl --user enable --now pingpulse`

### Windows
For v1: `pingpulse start` runs in foreground. Service registration via `sc.exe` deferred to v2.

## Cross-Platform Build

Target platforms:
- macOS: aarch64-apple-darwin, x86_64-apple-darwin
- Linux: x86_64-unknown-linux-gnu, aarch64-unknown-linux-gnu
- Windows: x86_64-pc-windows-msvc

Build via `cargo-zigbuild` or `cross` in CI. GitHub Actions workflow to be created as a separate task.

## Error Handling

- WebSocket errors → log, trigger reconnect
- Speed test errors → log error, send `error` message over WS
- Config write errors → log error, keep running with in-memory config
- Registration errors → print error to stderr, exit with code 1
- All panics caught at top level, logged, daemon restarts via service manager
